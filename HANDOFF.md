# 부부가계부 실전 데이터 이관 작업

## 배경

이 프로젝트는 이미 "Five Jars" 컨셉의 가계부 앱으로 동작 중이다. 이번 작업의 목적은 **실제 부부가 2025년 한 해 동안 애플 Numbers로 관리해 온 가계부**를 이 앱으로 옮기는 것이다. 엑셀 구조를 별도 Claude 세션에서 분석한 결과, 현재 앱 구조와 매우 잘 맞으며 **4가지 핵심 개념만 추가로 구현**하면 충분하다.

이 문서는 그 분석 결과와 작업 지시를 담고 있다. 우선 `src/` 아래 주요 파일들(특히 `types/index.ts`, `lib/db.ts`, `pages/Dashboard.tsx`, `pages/JarsPage.tsx`, `pages/AddTransaction.tsx`, `hooks/useStore.ts`)을 읽고 현재 구조를 파악한 뒤, 아래 순서대로 작업해 달라.

---

## 실제 가계부의 핵심 구조 (분석 결과)

### 1. 항아리 분배 — 이미 구현됨 ✅

실제 부부는 수입을 5가지로 분배 중이며, 현재 앱 구조와 동일하다:

| 항아리 | 실제 비율·금액 (2025년) | 비고 |
|---|---|---|
| 십일조 | 수입의 약 10% (변동) | 수입에 연동 |
| 헌금 | ¥8,000 고정 | |
| 투자 | ¥30,000 고정 | |
| 저축(A) | ¥20,000 고정 | |
| 쓰는 돈 | 나머지 전액 | 일일 지출의 예산 |

**현재 앱이 "비율 기반 자동 분배"라면, 일부 항아리는 고정 금액 기반이라는 점이 다를 수 있다.** `SettingsPage.tsx`와 `lib/db.ts`의 항아리 배분 로직을 확인해서, **항아리별로 "비율(%)" 또는 "고정 금액(¥)" 중 하나를 선택할 수 있게** 확장이 필요한지 판단해 달라. 이미 지원하면 이 작업은 스킵.

### 2. 3구간 지출 목표액 — **미구현, 핵심 추가 기능** ⚠️

이게 이 가계부의 가장 독창적인 부분이다. 한 달을 **3구간**으로 나눠 관리한다:

- 초반: 1–10일
- 중반: 11–20일
- 후반: 21–말일

각 구간별로 카테고리별 목표액을 설정하고, 실시간으로 잔여금을 보여준다. 월 단위 예산만으로는 "월초에 다 써버리는" 문제를 해결하지 못하므로, 이 3구간 체크포인트가 핵심 UX다.

### 3. 일일 지출 카테고리 — **구조 확인 필요**

실제 엑셀의 일일 지출은 다음 6개 카테고리로만 기록된다:

1. **식비 / 외식**
2. **식비 / 식료품**
3. **식비 / 간식**
4. **교통비**
5. **생필품**
6. **기타(특별이벤트)**

현재 앱의 카테고리 체계가 이것과 맞는지, 또는 "대분류(식비) / 소분류(외식·식료품·간식)" 2-depth 구조를 지원하는지 `types/index.ts`와 `AddTransaction.tsx`를 확인해 달라. 2-depth가 없다면 추가가 필요하다.

### 4. 공과금 격월 처리 — **미구현**

공과금은 5개 세부 항목으로 관리되며, **수도세만 격월 청구**다:

| 항목 | 패턴 | 월평균 |
|---|---|---|
| 수도세 | 격월 (1·3·5·7·9·11월) | ¥5,000대 |
| 전기세 | 매월, 계절 변동 큼 | 여름·겨울 ¥11,000대 |
| 가스세 | 매월 | ¥5,000–¥7,000 |
| 인터넷 | 매월 고정 | ¥4,180 |
| 통신비 | 매월 고정 | ¥4,359 |

공과금 계산 시 격월 항목은 해당 월에만 계상되지만, **예산 계획 시에는 월 평균으로 분산 반영**해야 "수도세가 청구되는 달만 초과 지출"로 보이는 오해를 피할 수 있다.

### 5. 특별지출 (여유분에서 처리) — **미구현**

아이폰 할부, 한국행 비행기, 사업비 같은 **이벤트성 대형 지출**을 별도 추적한다. 구조는:

- 이름 (예: "한국행 비행기")
- 총액 (예: ¥722,023)
- 누적 사용액 (할부금, 분할 결제 기록)
- 잔액
- 메모

일반 거래와 분리되어야 하며, 여러 달에 걸쳐 분할 지출되는 경우가 많다.

### 6. 개인 돈 vs 공동 지출 — **확인 필요**

공동 지출 외에 부부 각자에게 ¥25,000씩 "개인 용돈"이 지급된다. 개인돈으로 카드 사용 시 **"입금해야 하는 돈"**으로 추적한다(나중에 공동 계좌에 입금하기 위함).

현재 앱에 멤버 개념(`useActiveMember.ts`, `MemberSelectSheet.tsx`)이 있으므로, 거래에 **"결제자(한결/수민/공동)"** 필드가 있는지, 없다면 추가해야 한다.

## 작업 지시 (우선순위 순)

### Phase 1: 현재 구조 파악 및 갭 분석

다음 파일들을 **순서대로** 읽고 각 항목에 대해 현재 구현 상태를 보고해 달라. 코드를 수정하기 전에 이 단계를 먼저 완료할 것.

1. `src/types/index.ts` — `Transaction`, `Jar`, `Category` 타입이 어떻게 정의돼 있나?
2. `src/lib/db.ts` — IndexedDB 스키마(버전, 테이블, 인덱스)는?
3. `src/pages/AddTransaction.tsx` — 카테고리·소분류·결제자 필드가 있나?
4. `src/pages/JarsPage.tsx`, `src/pages/SettingsPage.tsx` — 항아리 분배 로직이 비율 기반인지 고정금액 지원하는지?
5. `src/pages/BudgetPage.tsx` — 예산 단위가 월 단위인지, 구간 단위도 지원하는지?
6. `src/hooks/useStore.ts` — 데이터 훅의 구조

보고 형식:
```
[항목] 현재 상태: ... / 갭: ...
```

### Phase 2: 데이터 모델 확장

Phase 1 결과를 바탕으로 아래 항목 중 **부족한 것만** 확장한다. IndexedDB 스키마 버전을 올리고 마이그레이션을 작성할 것.

```typescript
// 추가·확장이 필요할 수 있는 타입 스케치 (실제 네이밍은 기존 코드에 맞춰라)

type Transaction = {
  // ... 기존 필드
  mainCategory: "식비" | "교통비" | "생필품" | "기타";  // 대분류
  subCategory?: "외식" | "식료품" | "간식";              // 소분류 (식비일 때만)
  payer: "공동" | "한결" | "수민";                       // 결제자
  isPersonalMoney: boolean;                              // 개인돈 사용 여부
};

type PeriodBudget = {                     // 신규
  yearMonth: string;                      // "2025-01"
  period: "early" | "mid" | "late";       // 1-10 / 11-20 / 21-말
  category: string;
  targetAmount: number;
};

type UtilityBill = {                      // 신규
  yearMonth: string;
  water?: number;      // 격월
  electricity: number;
  gas: number;
  internet: number;
  telecom: number;
};

type SpecialExpense = {                   // 신규
  id: string;
  name: string;
  totalAmount: number;
  fundingSource: "bonus" | "reserve" | "monthly";  // 재원
  transactions: { date: string; amount: number; memo?: string }[];
};

type JarAllocation = {                    // 기존 확장 (필요 시)
  jarId: JarId;
  mode: "percentage" | "fixedAmount";
  value: number;  // 비율(%) 또는 고정금액(¥)
};
```

**중요**: 기존 데이터와의 호환성을 위해 새 필드는 모두 `optional`로 시작하고, 마이그레이션에서 기본값을 채울 것.

### Phase 3: UI 구현 (우선순위 높은 것부터)

**3.1 AddTransaction 확장 (최우선)**
- 대분류 → 소분류 2-depth 카테고리 선택 (식비일 때만 소분류 노출)
- 결제자 선택 (공동/한결/수민)
- 개인돈 토글

**3.2 Dashboard에 "구간 잔여금" 카드 추가**
현재 날짜로 현재 구간(초/중/후)을 자동 판별 → 해당 구간의 목표액 대비 잔여금을 메인 화면에 카드로 표시. 디자인 힌트:

```
┌─────────────────────────────┐
│ 중반 (11-20일)   D-3 남음   │
│                             │
│ ¥18,432 / ¥35,000           │
│ ████████░░░░░░  52%         │
└─────────────────────────────┘
```

**3.3 BudgetPage에 구간 목표액 설정 추가**
월 단위 예산과 **별도로** 3구간 × 카테고리별 목표액을 설정할 수 있는 UI. 기존 월 단위 예산은 유지.

**3.4 공과금 입력·표시 화면**
SettingsPage 또는 별도 페이지에서 월별 공과금 5개 항목 입력. 수도세는 "격월" 배지로 표시. 월평균 자동 계산.

**3.5 특별지출 페이지 (최후순위)**
별도 탭 또는 설정 내 진입. 이벤트성 지출 목록 + 각 이벤트의 누적 지출 내역.

### Phase 4: 기존 데이터 임포트 (선택)

사용자가 2025년 엑셀 데이터를 앱으로 옮기고 싶어할 가능성이 높다. CSV 업로드 → 앱 데이터로 변환하는 임포트 기능이 필요할 수 있다. **Phase 3까지 완료된 후 사용자에게 필요 여부를 확인**하고 진행할 것.

## 작업 원칙

1. **기존 네이밍·컨벤션 존중**: 이미 `JarId`, `useStore` 같은 컨벤션이 있으니 새 타입·훅도 같은 스타일로 작성
2. **i18n 유지**: 새 UI 문자열은 `ko.json`, `en.json`, `ja.json` 세 파일 모두에 추가 (사용자는 일본 거주자, 한국인 부부 — 한/일 번역 정확도 중요)
3. **IndexedDB ↔ Supabase 양쪽 반영**: 스키마 변경은 `lib/db.ts`와 `api/` 양쪽에 동기화
4. **shadcn/ui 컴포넌트 재사용**: 새 UI도 기존 디자인 시스템 안에서
5. **Capacitor 호환 유지**: 웹 전용 API 사용 금지 (File System 등 필요 시 Capacitor 플러그인 사용)

## 완료 기준

Phase 1~3 완료 시점에, 사용자가 앱에서 다음을 할 수 있어야 한다:

- [ ] 월초에 수입 입력 → 5개 항아리에 비율/고정금액 혼합으로 자동 분배
- [ ] 하루 지출을 식비(외식/식료품/간식)/교통비/생필품/기타로 기록, 결제자 지정
- [ ] 홈에서 오늘이 속한 구간의 목표액 대비 잔여금 확인
- [ ] 예산 설정에서 구간별 × 카테고리별 목표액 설정
- [ ] 공과금 5개 항목 월별 입력, 수도세 격월 처리
- [ ] 특별지출 별도 추적
